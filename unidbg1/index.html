准备工作

案例: https://www.wandoujia.com/apps/8059828/history_v153

登录 app 之后, 在请求任意列表页面的时候抓包可以看到在 Query String里面有加密参数sign;

直接打开 apk 后发现是加固后的;

使用 dexdump 脱壳;

再使用 dex2apk 重新组装成 apk后用 jadx 打开

直接使用 jadx 打开重新打包好的 apk 即可 jadx-gui repacked.apk

Java 层定位

直接在 jadx 中搜索 “sign”, 可以定位到可疑的加密位置:

使用 objection 进一步确认

使用 frida 打印一下入参, 确定一下传入的参数是什么, 之后方便固定参数进行逆向;

主动调用

主动调用的结果和之前 frida 打印的结果相同, 说明该算法中应该没有随机数的参与; 看这个 sign 的长度是一个 32 chars, 猜测是一个 md5 之后的结果; 用CyberChef 验证一下, 发现结果不对, 很可能是加盐了;

只能到 native 层继续查看了;

native 层定位

定位到加密接口, 跟进去看一下就可以定位到 native 层;

使用杨神的脚本, 对动态注册进行 hook 打印, 因为我在安装 apk 的时候指定了在 arm32 位下运行, 所以这个地址区别于 arm64 下的地址;

frida -U --no-pause -f com.sina.oasis -l hook_RegisterNatives.js

找到对应的 s 方法, 偏移为 0xc365; 函数签名([BZ]), 返回值为 String;

frida 版本切换到 14, ida 打开目标 so, 跳转到 0xc365; 直接使用 traceNatives 进行 trace, 然后主动调用, 获取到调用的路径;

frida-trace -UF -O /Users/zhangyang/Downloads/lvzhou3.5.8/lib/armeabi-v7a/liboasiscore_1628839431.txt

之前就怀疑是 md5, 可以使用 findhash 进行可疑的函数定位, 方便快速找到加密位置;

在 0x8975 地址, 在主动调用过程中 traceNatives 经过了该方法, findhash 中判断该方法中有魔数初始化, 我们重点关注这个方法, 最终可以在该方法调用 sub_8AB2 的时候进行 hook, 获取到入参和结果;

invoke 的结果为: 01ccabfc1f50dc3d4db1dd9fcd7e6910

观察 hook 的结果, 可以看到入参的盐为固定的: YP1Vty&$Xm*kJkoR,Opk&

md5 之后的结果在 arg0中可以找到:

使用 CyberChef 确认一下, 结果是正确的;
